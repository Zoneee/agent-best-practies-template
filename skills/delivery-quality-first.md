# delivery-quality-first

## 目的
无论处于 Plan、Agent 还是 Ask 模式，先把"如何保证安全交付、如何验证、如何收集证据、如何控制风险"说清楚，再讨论具体修什么或怎么做。

过程控制优先于修复叙述。

## 适用范围
- 制定或审阅执行计划
- 提供方案建议或技术咨询
- 执行实现任务
- 评审或复盘交付质量

## 强制输出顺序
1. 交付目标与成功标准
2. 范围边界与非目标
3. 质量门禁
4. 验证方式与证据来源
5. 风险、回滚与止损
6. 具体修复项 / 实现步骤 / 建议方案

若第 6 项比前 5 项更完整、更具体、更有可执行性，应视为输出结构失衡，需要重写。
判断参考：质量控制部分（前 5 项）合计的条目数和具体程度，应不少于修复/实现部分（第 6 项）。

## Plan 模式要求
- 计划必须先说明如何保证交付质量，再写问题修复方案。
- 必须显式列出验证计划、证据要求、风险与回滚策略。
- 若无法定义验证方式，不得将计划标记为可执行。

## Agent 模式要求
- 开始执行前，先声明本次采用的质量门禁与验证路径。
- 不得跳过范围控制与证据设计直接开始改动。
- 若信息不足，先补计划、文档或约束，再继续执行。

## Ask 模式要求
- 不得只回答"推荐做什么"，必须同时回答"为什么这样做可以被验证"。
- 必须说明建议的适用边界、主要风险和下一步验证动作。
- 若证据不足，应优先暴露不确定性，而不是给出过度确定的结论。

## 自检问题
- 我是否先写了质量控制，再写了修复内容？
- 我是否写清楚了范围边界与非范围？
- 我是否写清楚了验证方式与证据来源？
- 我是否说明了风险、回滚与止损方式？
- 我的输出是否在 Plan / Agent / Ask 任一模式下都能成立？
