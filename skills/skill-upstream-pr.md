# Skill：受控回流 PR 工作流

## 目的
为下游项目提供一套标准化机制，用于将经过验证的共性 Skill 改动整理成面向模板主仓的 Pull Request 提案。

**重要声明：这是"受控回流 PR 工作流"，不是自动反向同步。**  
下游改动不会自动写入模板主仓。本 Skill 的目标是帮助下游项目整理、判断、生成一份高质量的上游回流 PR 提案，由人工审批后决定是否合并。

## 适用场景
- 下游项目在实践中积累了对某个 Skill 的改进，认为该改进具有普遍适用价值
- 下游项目新增了一个 Skill，希望评估它是否适合纳入模板主仓的 `base` 层
- 下游项目发现模板主仓中某个 Skill 的表述不准确或存在缺失，需要提议修正
- 跨多个下游项目都出现了类似需求，具备抽象为通用 Skill 的条件

## 不适用场景
- 项目专属业务逻辑（与具体业务领域强绑定）
- 项目特定技术栈约束（如某数据库、某框架的专有规则）
- 尚未在实际任务中验证过的想法或草稿
- 仅在单个项目中出现过一次的模式，尚无法判断通用性
- 任何包含项目机密、内部代号或业务敏感信息的内容

## 回流前判断

在准备回流前，请逐一确认以下问题：

1. **是否具有共性？**  
   这个 Skill 改动是否在多个不同类型的项目中都适用，还是只对当前项目有价值？

2. **是否已验证？**  
   这个改动是否经过了至少一个完整任务周期的实际验证？是否有可引用的证据？

3. **是否已剥离项目专属内容？**  
   所有业务名称、内部系统引用、项目特定路径是否已替换为通用描述或占位符？

4. **是否与现有 Skill 冲突？**  
   模板主仓是否已有表达同一意图的 Skill？若有，是替换还是补充？

5. **是否有破坏性变更？**  
   改动是否会影响已有下游项目的行为？是否需要迁移指导？

## 回流目标选择

根据以下标准决定回流目标：

### 回流到 `base`（通用层）
- Skill 适用于所有项目类型（backend / frontend / library 等）
- 不依赖特定技术栈或运行环境
- 改动经过多种项目类型验证

### 回流到特定 `overlay`（叠加层）
- Skill 适用于某类项目（如 backend-service overlay）
- 依赖特定技术栈或架构模式，但在该类型内具有通用性
- 注明目标 overlay 名称（如 `overlays/backend-service`）

### 不适合回流（建议本地保留）
- Skill 与当前项目业务强相关
- 改动依赖项目特定基础设施
- 改动仅在当前项目的特定阶段有效
- 无法在不暴露项目内部信息的情况下通用化

## 回流 PR 必备内容

一份高质量的上游回流 PR 提案必须包含以下内容：

1. **来源声明**：来自哪个下游项目、项目类型、基于的模板版本
2. **变更摘要**：改动了哪个 Skill、改动了什么、为何改动
3. **回流理由**：为何认为该改动应纳入模板主仓，而不仅仅保留在下游
4. **回流目标**：明确指出是 `base` 还是某个 `overlay`
5. **验证证据**：该改动在下游项目中的验证结果（任务名称、时间、结论）
6. **非回流内容**：明确说明哪些内容被剥离、为何剥离
7. **风险与兼容性**：是否有破坏性变更，其他下游项目需注意什么

## 生成产物

使用 `propose-skill-upstream-pr.sh` 脚本（或手动整理）生成以下产物：

```text
output/
├── metadata.json              # 来源、版本、项目类型等元信息
├── proposal.md                # 回流提案说明（符合 PR 模板格式）
├── pr-body.md                 # 可直接用于发起 PR 的正文草稿
└── patches/
    └── skills/<skill-name>.md # 候选回流文件（已剥离项目专属内容）
```

**注意**：生成产物不会自动创建跨仓库 PR。需由人工审阅后，通过正常 GitHub PR 流程向模板主仓发起。

## 自检问题

在发起上游 PR 提案前，完成以下自检：

- [ ] 是否确认该改动不包含任何项目专属信息？
- [ ] 是否确认该改动在实际任务中经过验证，有证据可引用？
- [ ] 是否明确说明了回流目标（base / overlay）及选择原因？
- [ ] 是否列出了非回流内容，并说明了剥离原因？
- [ ] 是否评估了对其他下游项目的潜在影响？
- [ ] PR 正文是否符合 `templates/upstream-skill-pr-template.md` 格式？
- [ ] 是否已在本地通过 dry-run 预览生成产物并确认无误？
